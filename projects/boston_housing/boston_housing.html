Implementation: Define a Performance Metric¶

It is difficult to measure the quality of a given model without quantifying its performance over training and testing. This is typically done using some type of performance metric, whether it is through calculating some type of error, the goodness of fit, or some other useful measurement. For this project, you will be calculating the coefficient of determination, R2, to quantify your model's performance. The coefficient of determination for a model is a useful statistic in regression analysis, as it often describes how "good" that model is at making predictions.

The values for R2 range from 0 to 1, which captures the percentage of squared correlation between the predicted and actual values of the target variable. A model with an R2 of 0 is no better than a model that always predicts the mean of the target variable, whereas a model with an R2 of 1 perfectly predicts the target variable. Any value between 0 and 1 indicates what percentage of the target variable, using this model, can be explained by the features. A model can be given a negative R2 as well, which indicates that the model is arbitrarily worse than one that always predicts the mean of the target variable.

For the performance_metric function in the code cell below, you will need to implement the following:

Answer: The "Grid Search" is a technique for tuning the hyperparameters of a ML algorithm. When an algorithm accepts more than one hyperparameter we may want to test more than one values to each hyperpsrameter. To make this possible actually we need to create a "grid" which each cell is a different combination of the hyperparameters' values.
For example a grid of two hyperparameters with 3 values each would look somthing like this:

n_estimators n_estimators n_estimators
max_depth 1 / 1 1 / 10 1 / 100
max_depth 10 / 1 10 / 10 10 / 100
max_depth 100 / 1 100 / 10 100 / 100

In a generalized form we can create a numHP dimensional table with

\begin{equation*} \prod_{i=1}^{(numHP)}(numValues)_i \end{equation*}

cells / combinations

By applying this method we test algorithm's score across all the different combinations and we select the combination with the best score.

Note: numHP: number of hyperparameters, numValues: number of values of a hyperparameter

Answer: In k-fold cross validation we split randomly the data (usually the training data) in k equal parts. Then we perform k iteration where each one of the folds is used for validation and the rest for training. During the grid search we test all the available hyperparameter combinations on the testing data but if we use the same data to evaluate the results there is the danger of overfitting on them and lead us to the wrong values.
To avoid overfitting we usually split the training data to training data and validating data but sometimes splitting the to three parts (training, validating, testing) is not possible (small dataset) or doing so minimizes significantly the amount of training data. With the use of k-fold cross validation during the hyperparameter tuning, we can use the training data both for training and validating without compromising the hyperparameters selection due to overfitting.

# TODO: Import 'make_scorer', 'DecisionTreeRegressor', and 'GridSearchCV'

from sklearn.metrics import make_scorer
from sklearn.tree import DecisionTreeRegressor
from sklearn.model_selection import GridSearchCV


def fit_model(X, y):
    """ Performs grid search over the 'max_depth' parameter for a 
        decision tree regressor trained on the input data [X, y]. """

    # Create cross-validation sets from the training data
    # sklearn version 0.18: ShuffleSplit(n_splits=10, test_size=0.1, train_size=None, random_state=None)
    # sklearn versiin 0.17: ShuffleSplit(n, n_iter=10, test_size=0.1, train_size=None, random_state=None)
    cv_sets = ShuffleSplit(n_splits=10, test_size=0.20, random_state=0)

    # TODO: Create a decision tree regressor object
    regressor = DecisionTreeRegressor()

    # TODO: Create a dictionary for the parameter 'max_depth' with a range from 1 to 10
    params = {'max_depth': list(range(1, 11))}

    # TODO: Transform 'performance_metric' into a scoring function using 'make_scorer'
    scoring_fnc = make_scorer(performance_metric)

    # TODO: Create the grid search cv object --> GridSearchCV()
    # Make sure to include the right parameters in the object:
    # (estimator, param_grid, scoring, cv) which have values 'regressor', 'params', 'scoring_fnc', and 'cv_sets' respectively.
    grid = GridSearchCV(regressor, params, scoring_fnc, n_jobs=-1, cv=cv_sets)

    # Fit the grid search object to the data to compute the optimal model
    grid = grid.fit(X, y)

    # Return the optimal model after fitting the data
    return grid.best_estimator_

Answer:
The constructed model should not be used in real-world settings for a number of reasons.
The most obvious is that the data is outdated. The cumulative inflation since 1978 is 272.58% this means that a house with an average price (\$454,343) today would have a price of $1,692,798 (source: inflationdata.com).
Additionally, even if we adjust the prices taking inflation into account, the possibility that the "socioeconomic" variables ('LSTAT' and 'PTRATIO') remain the same is minor.
There are some other factors that make this model unusable in real-world:

  • Supply and Demand. Most probably both supply and demand have not remained the same after 39 years.
  • Location. A model trained with data from a specific area cannot be deployed in another area. As an example, the average price per square foot in Washington, D.C. is \$497 while in Indiana is \$86 (source: GoBankingRates)
  • Total Area. Although the number of rooms is a good indicator of the floor area of the house, we cannot estimate the value of a house without this information. The model will produce the same prediction for both a house with small rooms and for one with big rooms.
  • Amenities There are amenities that affect the price, like parking space, number of bathrooms, gardens, pools etc. missing from the dataset.
  • House Condition. The construction year and any upgrades since then affect the price very significantly.

In general, the model could be used as an indication of the specific area's prices, for the time the prices collected but this is all it can offer.
